## 2. 핵심 요약 (Summary)

### Virtual Memory란?
- **실제 메모리에 프로그램 전체를 올리지 않고도 실행 가능**하게 하는 기법
- 물리 메모리보다 큰 프로그램도 실행 가능
- 논리 메모리(logical memory)와 물리 메모리(physical memory)를 분리해 **추상화된 메모리 공간 제공**
- **파일 및 라이브러리 공유** 및 **프로세스 생성 효율화** 지원

---

### Virtual Address Space (가상 주소 공간)
- 프로세스 기준 **논리적 메모리 구조**
- 보통 address 0 부터 연속적으로 존재하는 것처럼 보임
- 실제 물리 메모리와는 다름
- 여러 프로세스가 같은 라이브러리를 **페이지 공유**(page sharing) 방식으로 사용 가능

---

### Demand Paging (요구 페이징)
- 프로그램 전체를 메모리에 올리지 않고, **실행 시 필요한 페이지만 적재**
- 페이지 유효 여부를 **valid-invalid bit**로 구분
  - valid: 메모리에 존재
  - invalid: 아직 메모리에 없음(디스크에 있음)

- **Page Fault 처리 절차**
  1. 접근 가능한 지 확인
  2. 유효하면 **free frame** 확보
  3. 디스크에서 해당 페이지 읽어오기
  4. 페이지 테이블 갱신
  5. 인터럽트된 명령어 재시작
- **Pure Demand Paging**: 시작 시 아무 페이지도 올리지 않고 첫 실행에서 page fault 발생하며 점진적으로 로딩
- 성능은 **Locality of Reference**(참조 지역성) 덕분에 현실적으로 양호
  - Locality of Reference: 프로그램이 메모리를 참조할 때 **특정 영역을 집중적으로 반복해서 접근하는 경향**
  - Temporal locality: 시간적으로 최근 접근한 데이터를 다시 사용
  - Spatial locality: 공간적으로 가까운 주소를 함께 참조

---

### Copy-on-Write (COW)
- `fork()` 시 부모, 자식이 페이지를 공유하다가, **쓰기 동작 발생 시점에 복사**(copy)

---

### Page Replacement (페이지 교체)
- free frame이 없을 경우, **사용하지 않는 프레임을 디스크로 내보내고 새로운 페이지 적재**
- **Page Replacement 알고리즘**
  - **FIFO**: 가장 먼저 들어온 페이지 제거. Belady’s Anomaly 발생 가능
  - **Optimal** (OPT): 앞으로 가장 오래 안 쓸 페이지 제거. 이론적 최적
  - **LRU** (Least Recently Used): 가장 오랫동안 사용되지 않은 페이지 제거
    - 과거의 참조 패턴(최근 사용 여부)을 근거로 미래 접근을 예측해서 locality를 활용(최근 사용 -> 앞으로도 사용될 가능성 높음)
    - Counter 방식, Stack 방식, Reference bit 기반 근사 알고리즘(Second-Chance, Clock 등)
- Belady’s Anomaly
  - 일반적으로는 프레임 수가 늘어나면 page fault가 줄어야 정상인데, FIFO 알고리즘에서는 프레임 수를 늘렸는데도 오히려 page fault가 증가하는 현상

| 알고리즘                                  | 특징                                            | 장점                              | 단점                               | Belady’s Anomaly |
| ------------------------------------- | --------------------------------------------- | ------------------------------- | -------------------------------- | ---------------- |
| **FIFO** (First-In First-Out)         | 가장 먼저 들어온 페이지부터 교체                            | 구현 간단, 직관적                      | 오래된 페이지가 자주 쓰이는 경우에도 교체됨         | 발생 가능          |
| **OPT** (Optimal / MIN)               | 앞으로 가장 오랫동안 사용하지 않을 페이지 교체                    | Page Fault 최소, 이론적 최적           | 미래 참조를 알아야 함 -> 실제 구현 불가 (비교/연구용) | 없음             |
| **LRU** (Least Recently Used)         | 가장 오랫동안 사용하지 않은 페이지 교체                        | locality 반영 -> 성능 우수, 실무에서 자주 사용 | 시간, 하드웨어 지원 필요 (카운터, 스택 등)        | 없음             |
| **Second-Chance** (Clock)           | FIFO + reference bit 사용, 참조된 페이지에 한 번 더 기회 부여 | LRU 근사 구현, 성능 양호                | 구현 복잡, 정확한 LRU는 아님               | 없음             |
| **LRU Approximation** (Reference Bit) | 페이지 접근 시 bit=1, 교체 시 0인 페이지 우선                | 하드웨어 부담 줄임                      | 정확성 떨어짐, 근사치                     | 없음             |

### Base/Bound vs Paging
| 구분                 | **Base/Bound 방식**                        | **페이징 방식**                                                              |
| ------------------ | ---------------------------------------- | ----------------------------------------------------------------------- |
| **기본 아이디어**        | 프로세스마다 시작 주소(Base)와 크기(Bound)를 지정해 보호    | 프로세스 주소 공간을 고정 크기 \*\*페이지(Page)\*\*로 나누고, 물리 메모리 \*\*프레임(Frame)\*\*에 매핑 |
| **주소 변환**          | 물리 주소 = Base + 논리 주소                     | 물리 주소 = \[프레임 번호 + 페이지 오프셋]                                             |
| **보호(Protection)** | Bound 넘어가면 Trap 발생 → 프로세스 격리             | 페이지 테이블의 접근 권한 비트(R/W/X 등)로 제어                                          |
| **단편화**            | **외부 단편화 발생** (프로세스 크기가 다양하므로 빈 공간이 흩어짐) | **외부 단편화 없음**, 대신 마지막 페이지에 **내부 단편화** 발생                                |
| **장점**             | 구현이 단순, 레지스터 2개로 가능                      | 메모리 활용 효율 높음, 큰 주소 공간 지원, 현대 HW 친화적                                     |
| **단점**             | 외부 단편화 심각, 메모리 압축 필요                     | 페이지 테이블 관리 비용, 주소 변환 오버헤드 (TLB로 완화)                                     |
| **현대 활용 여부**       | 옛날 시스템에서만 사용 (단순 보호 목적)                  | 사실상 모든 현대 OS가 사용 (가상 메모리 기반)                                            |

### Frame Allocation (프레임 할당)
- **Equal allocation**: 모든 프로세스에 동일한 수의 프레임 할당
- **Proportional allocation**: 프로세스 크기에 비례하여 할당
- Global vs Local Replacement
  - Global: 전체 시스템 프레임 중 교체
  - Local: 자기 프레임 내에서만 교체

---

### Swap 
- 메모리가 부족할 때, 프로세스나 페이지를 디스크(보조 저장장치)에 잠시 내보내고 필요할 때 다시 가져오는 것
- 프로세스 단위 스왑 (과거 방식)
  - 전체 프로세스를 통째로 디스크에 내보냄
  - 메모리에 다시 올릴 때는 프로세스 전체를 로딩해야 해서 오버헤드가 큼
- 페이지 단위 스왑 (현대 방식)
  - 가상 메모리와 결합되어 필요한 페이지 단위만 디스크에 내보냄
  - 페이지 폴트가 나면 다시 디스크에서 읽어와 RAM에 올림


---

### Thrashing (스래싱)
- 프로세스가 페이지 교체에만 몰두해 실제 일은 거의 못하는 상태
- 원인: 프로세스에 필요한 프레임 수 부족 -> page fault 과다 발생

---

### Working-Set Model (작업 집합 모델)
- locality 기반 접근
- 최근 n개 참조 내에 포함된 페이지 집합을 working set이라 정의
- working set 크기 = 프로세스 실행에 필요한 최소 프레임 수 추정
- 이를 바탕으로 프레임을 적절히 할당해 thrashing 방지

### 스터디 질문 정리
#### 세그멘테이션과 페이징의 차이를 설명해주세요
- Segmantation: 코드, 데이터, 스택처럼 **논리적 단위**로 메모리를 나눔, 크기가 가변적, 프로그램 구조를 반영하므로 관리가 직관적이지만, 외부 단편화가 생기기 쉬움
- Paging: **고정 크기**의 페이지로 쪼개어 프레임에 매핑, 외부 단편화는 줄이지만, 마지막 페이지에서 내부 단편화가 생길 수 있음
- 외부 단편화(External Fragmentation): 메모리를 여러 블록으로 나눠 쓰다 보면, 총 남은 공간은 충분한데도 연속된 큰 블록이 없어서 할당을 못하는 상황. 즉, 빈 공간이 여기저기 흩어져 있어서 못 쓰는 낭비가 생기는 상황
- 내부 단편화(Internal Fragmentation): 고정 크기 블록(예: 페이지, 프레임)을 할당했는데, 실제로는 그보다 작은 데이터만 쓰는 경우 생기는 낭비. 즉, 할당된 블록 안에서 남는 공간이 낭비.

#### TLB(Translation Lookaside Buffer)란 무엇이고 왜 사용하는가?
- 최근 접근한 가상주소 -> 물리주소 변환 결과를 저장하는 캐시
- 페이지 테이블 접근을 매번 메모리에서 하면 느리므로, TLB가 있으면 **주소 변환 속도가 크게 향상**시킴
- Hit율이 높을수록 실제 메모리 접근 시간(EAT)이 줄어듦

#### paging 기법에서 특정 page에 접근할 때 어떤 과정이 일어나는지 설명하세요
- CPU는 논리 주소를 page number와 offset으로 나눕니다. page number로 페이지 테이블을 찾아 frame number를 얻고, offset과 합쳐 최종 물리 주소(RAM)를 만듭니다. 이때 TLB에 page number가 있으면 바로 변환되고, 없으면 메모리에 있는 페이지 테이블을 읽어야 하므로 더 느립니다.

#### 클락 알고리즘에 대해서 설명하세요
- LRU를 근사한 페이지 교체 방식
- 모든 페이지 프레임을 원형 큐(시계 모양)에 배치하고, 각 페이지마다 reference bit를 둡니다. 시계바늘(hand)이 한 칸씩 이동하며 교체 대상을 찾습니다. 바늘이 가리킨 프레임의 reference bit이 0이면 최근에 사용되지 않았다는 뜻이라 교체합니다. 만약 1이면 최근에 사용됨을 의미하므로 bit을 0으로 바꾸고 다음으로 넘어갑니다. 결국 한 바퀴 돌면 오래 참조되지 않은 페이지는 bit=0 상태가 되어 교체됩니다.
- FIFO보다 **locality를 더 반영**하면서 구현이 단순
  
#### 최신 접근 흔적(reference bit)만으로는 왜 정확한 LRU를 대체하기 어렵나요?
- reference bit은 최근에 접근 여부만 알 수 있고, 얼마나 오래 사용되지 않았는지 순서를 알 수 없기 때문
- 예를 들면, 단순히 1/0만 기록한다치면, 여러 페이지가 다 1이면 누가 더 오래된지 모름
- 이를 방지하기 위해, 클락 알고리즘을 사용해서 일정 시간 접근이 없는 페이지 교체

#### Clock 핸드 회전 속도/핸드 수 조절은 어떤 효과가 있나요?
- Clock 알고리즘에서 바늘이 도는 속도를 조절하면, 페이지가 교체 대상이 되기 전에 더 오래 메모리에 남을 수 있음
  - 방금 접근한 페이지가 교체되지 않고 남아있는 시간이 늘어나 페이지 폴트가 줄어들 수 있음
- 핸드가 여러 개 있으면 최근 참조된 페이지에 더 유연하게 기회를 줄 수 있음
  - 다중 핸드(Multi-Hand) 기법: 한 페이지가 교체 후보로 표시된 후에도 여러 바늘이 지나가는 과정에서 또 참조될 기회가 생김(여러 개의 핸드를 시차를 두고 돌림 -> 전 핸드에서 0으로 바꼈더라도 다음 핸드 동작 전에 해당 페이지가 참조되어 bit 1 로 세팅될 수 있음) -> 짧은 주기로 재사용되는 데이터에 적절
- 즉, locality 패턴에 맞춰 tuning하면 page fault율을 줄일 수 있음

#### 멀티 레벨 페이지 테이블에 대해서 설명하세요
- 주소 공간이 커질수록 페이지 테이블 크기가 너무 커짐 -> 멀티 레벨 페이지 테이블
- 상위 테이블이 하위 테이블을 가리키는 트리 구조
- 필요한 테이블만 메모리에 유지할 수 있어 공간 효율성이 좋지만, 주소 변환 시 메모리 접근 횟수가 늘어나는 비용 발생
- 페이지 테이블 커지면 발생하는 문제
  - 연속된 배열 구조이기 때문에 모든 페이지를 메모리에 올리게 됨 -> 메모리 낭비
  - TLB에 모든 정보를 담기 어려움 -> TLB miss 증가
  - Context switch 비용 증가

#### 연속 할당과 비교하여 페이징이 외부 단편화를 어떻게 해소하는지, 대신 생기는 비용은 무엇인지 설명하세요.
- 연속 할당은 큰 블록을 찾기 어려워 외부 단편화가 발생(free 하는 순간에 순서가 보장되지 않기 때문에 공간이 사이사이 발생함) -> 해결 방법: 주기적으로 합침(성능 포기)(compaction)
- 페이징은 물리 메모리를 고정 크기 블록으로 쪼개 쓰므로 외부 단편화 문제 방지
- 대신 마지막 페이지에 빈 공간이 생겨 내부 단편화가 생기고, 페이지 테이블 관리, TLB 미스 등의 오버헤드가 발생

#### 컨텍스트 스위치 시 TLB 미스가 왜 증가하고, 이를 줄이기 위한 HW/SW 기법은 무엇이 있나요?
- 프로세스가 바뀌면 주소 공간도 바뀌므로 기존 TLB 엔트리가 무효화
- 따라서 새 프로세스 실행 시 초기 접근에서 TLB miss가 많이 발생
- HW: ASID/PCID 같은 프로세스 식별자를 TLB 엔트리에 두어 스위치 후에도 재사용
- SW: 커널이 자주 쓰는 페이지를 미리 고정

#### 가상 메모리의 목적과 참조 지역성(locality)이 왜 성능을 성립시켜 주는지 설명하세요.
- 물리 메모리보다 큰 프로그램도 실행 가능하게 함
- **프로세스마다 독립적인 주소 공간**을 제공(logical address, physical address 분리 / 프로세스마다 페이지 테이블 다름)
  - 보호: 한 프로세스가 다른 프로세스 메모리 침범 못하게
  - 편의성: 실제 물리 메모리 주소를 몰라도 연속적이고 큰 메모리를 사용하는 것처럼 개발 가능하게
  - 효율성: 물리보다 큰 프로그램 실행, 여러 프로세스 동시 실행
- 실제 성능이 나오는 이유는 locality 때문
- 프로그램은 시간적으로 최근 접근 데이터를 다시 쓰거나, 공간적으로 인접 데이터를 연속 접근하는 경향이 있어, 일부 페이지만 메모리에 두어도 대부분의 접근을 커버할 수 있음

#### MMU 랑 페이지 테이블 차이
- MMU (Memory Management Unit)
  - CPU 안(혹은 메모리 관리 전용 하드웨어)에 있는 하드웨어 장치
  - CPU가 낸 가상 주소 -> 물리 주소로 변환
  - 변환 과정에서 TLB 캐시를 먼저 확인하고, miss가 나면 페이지 테이블을 참조
  - 주소 변환을 실제로 실행하는 하드웨어

- 페이지 테이블 (Page Table)
  - OS가 프로세스마다 유지하는 **데이터 구조(메모리 상에 존재)**
  - 가상 페이지 번호 -> 물리 프레임 번호 매핑 정보 + 보호 비트(R/W/X), 유효 비트 등
  - MMU가 주소 변환할 때 참조하는 자료구조
  - 주소 변환을 위한 매핑 정보를 저장하는 테이블

#### 가상 -> 물리 주소 변환 흐름
1. CPU
   - 가상 주소(Virtual Address = [page number | offset])를 생성
2. MMU (Memory Management Unit, 하드웨어)
   - 변환을 실제로 수행하는 장치
   - 먼저 TLB를 확인
3. TLB (Translation Lookaside Buffer, 캐시)
   - 최근 사용한 가상 페이지 번호 -> 물리 프레임 번호 매핑 캐시
   - TLB hit -> 바로 물리 프레임 번호 얻음 -> 5번 바로
   - TLB miss -> 페이지 테이블 참조
4. 페이지 테이블 (Page Table, 메모리에 있는 자료구조)
   - OS가 프로세스별로 관리
   - 해당 가상 페이지 번호에 대응하는 물리 프레임 번호를 가져옴
   - MMU가 이를 TLB에 적재해 캐싱
5. 최종 물리 주소(Physical Address)
   - [물리 프레임 번호 | offset]으로 조합
   - 실제 RAM에 접근
